Goal: drastically reduce Google Places API calls by introducing a 3-layer cache and a pre-seeded Bay Area Places Index (4.5★+ and ≥100 reviews for restaurants/cafes/bars/desserts). Build the following exactly.

Context (do not change)

Frontend: React + TS + Vite + shadcn/Tailwind.

Backend: Express + TS.

DB: Postgres (Neon) + Drizzle ORM.

Existing tables: groups, members, activities, itineraries, itinerary_items, group_collections, category_search_history, user_profiles, etc.

Env: DATABASE_URL, OPENAI_API_KEY, GOOGLE_PLACES_API_KEY, optional GOOGLE_PLACES_API_KEY_2, NODE_ENV.

High-level tasks

Add 3-layer caching

Layer A: In-process LRU cache with TTL (12h default).

Layer B: Postgres cache tables (see schema).

Layer C: Client memoization (localStorage) for recent queries (frontend—basic key/value by location+filters).

Seeded Bay Area Index

Create a grid over the Bay Area (simple ~2–3 km spacing; a static array of lat/lng + radius is fine).

For each cell and type (restaurant, cafe, bar, bakery), call Nearby/Text Search once, follow next_page_token.

Keep only places with rating ≥ 4.5 and user_ratings_total ≥ 100.

Upsert into places table; then fetch Place Details (Basic fields only) and store.

Store photo_reference strings (do not download images).

Run this seed job manually (script) and set up a nightly cron (Replit “always on” task or simple scheduled runner if available).

Runtime query path

/api/places/search: DB-first by proximity + types. If results < threshold (e.g., 40), do one Google call as fallback, then cache.

Result set should include a small % of novelty (see below).

Novelty / “NEW” logic

Add 10–20% of options from adjacent cells or under-explored categories for that group.

Weekly refresh window: mark new additions as "NEW (this week)".

Cost guardrails

Centralize Google client with rate limit/token bucket, retries with jitter, round-robin across GOOGLE_PLACES_API_KEY[_2].

TTLs: Place Details 30 days; search buckets 24 hours (configurable).

Inflight request coalescing: dedupe concurrent calls to same cache key.

File changes (create/update)

Drizzle schema (add exactly)
// server/db/schema/places.ts
import { pgTable, text, integer, timestamp, numeric, jsonb } from "drizzle-orm/pg-core";

export const places = pgTable("places", {
  placeId: text("place_id").primaryKey(),
  name: text("name").notNull(),
  lat: numeric("lat").notNull(),           // store as numeric for Postgres; cast in app
  lng: numeric("lng").notNull(),
  types: text("types").array().notNull(),  // e.g., ['restaurant','food','point_of_interest']
  priceLevel: integer("price_level"),
  rating: numeric("rating"),
  userRatingsTotal: integer("user_ratings_total"),
  address: text("address"),
  phone: text("phone"),
  website: text("website"),
  googleMapsUri: text("google_maps_uri"),
  photoRefs: text("photo_refs").array(),
  softClosed: integer("soft_closed").default(0), // 0/1 flag if users report closed
  lastDetailsFetch: timestamp("last_details_fetch").defaultNow().notNull(),
  details: jsonb("details") // raw detail subset; avoid schema churn
});

export const placeSearchCache = pgTable("place_search_cache", {
  cacheKey: text("cache_key").primaryKey(), // e.g., 'cell:abc123:type:restaurant'
  placeIds: text("place_ids").array().notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
  ttlSeconds: integer("ttl_seconds").notNull()
});


Add indexes via migrations:

GIN on places.types

B-tree on (rating DESC, user_ratings_total DESC)

LRU + inflight coalescing (implement)

Export cached<T>(key, fetcher, ttlMs):

Check LRU; if miss, if an inflight promise for key exists return it; else call fetcher, set LRU with TTL, clear inflight, return.

Default LRU TTL: 12h. Allow per-call override.

Google client (implement)

Functions:

textSearch({query, location, radius, type, paginated: true})

nearbySearch({location, radius, type, paginated: true})

getPlaceDetails({placeId, fields: basic})

getPhotoUrl({photoRef, maxwidth}) (returns signed URL parameters per API rules)

Rate limiting: simple token bucket (X/min, Y/day), with exponential backoff and jitter on 429/5xx.

Key rotation: alternate between GOOGLE_PLACES_API_KEY and GOOGLE_PLACES_API_KEY_2 if both present.

Seed job logic (implement server/jobs/seed-bayarea.ts)

Iterate gridCells × types = ['restaurant','cafe','bar','bakery'].

For each cell/type:

Use Nearby (preferred) with sensible radiusM (2km urban, up to 5km suburban).

Page through results via next_page_token.

Filter: rating >= 4.5 AND user_ratings_total >= 100.

Upsert to places (basic fields).

Batch Details (basic) for those place_ids; update fields, photoRefs, lastDetailsFetch.

Save placeSearchCache row: cacheKey = cell:{id}:type:{type}, placeIds, ttlSeconds = 7*24*3600.

Idempotent upserts (on conflict—update non-PK fields).

Log counts: fetched/filtered/upserted/detailsFetched per cell.

/api/places/search (implement)

Query params: lat, lng, radius (m; default 3000), types (csv; default restaurant), limit (default 60).

DB-first:

Find places within radius (Haversine or PostGIS extension if available; if not, approximate by bounding box then filter).

Filter to requested types; order by (rating DESC, user_ratings_total DESC).

If results < 40:

Do one Google search (Text or Nearby), upsert, merge & dedupe, then return.

Novelty:

Inject 10–20% from adjacent grid cells or under-used cuisines for that group (hook left as TODO but expose slot).

Add isNew: true if createdAt (or first-seen) within last 7 days or not in group’s historical picks.

Response fields: minimal card needs + photoRefs[0] for images.

Frontend memo (lightweight)

Before calling the API, compute a key = v1:{lat_2dp}:{lng_2dp}:{radius}:{typesSorted}.

If a cached value exists from the last 2h, return it immediately while also re-fetching in background to refresh (stale-while-revalidate feel).

Admin seed route (optional)

POST /api/admin/seed/run guarded by NODE_ENV!=='production' or a secret header.

Kicks off seed-bayarea.ts job; returns status.

Acceptance criteria

On cold start, most user searches return DB-only results (no Google calls).

/api/places/search returns 60 results in under 300ms when served from DB/LRU on Replit.

One-click seed job produces ~2–5k Bay Area places meeting thresholds.

Google Places daily calls reduced by >70% vs. current.

Keys never exceed rate limits; if budget hits zero, system gracefully degrades to DB-only with a clear log warning.

New data shows up weekly with "NEW (this week)" tags without blowing quota.

Notes / Non-goals

Do not scrape Google; only use official Places endpoints per ToS.

Photos: keep photo_reference only; generate URLs on demand.

Future nice-to-have: Use OpenStreetMap to seed candidates then enrich via Google Details on demand.
